# Sentinel

Sentinel is the inbound protection layer. It validates AI-generated code before it reaches your codebase.

## The problem

AI generates code fast. That code can contain:

- Security vulnerabilities (SQL injection, XSS, command injection)
- Copyleft-licensed code in your proprietary project
- Deprecated or dangerous APIs
- Patterns that violate your team's standards

You catch these in code review. Sometimes. Sentinel catches them every time.

## What Sentinel does

**Security scanning**
- SQL and NoSQL injection patterns
- Command injection
- Cross-site scripting (XSS)
- Insecure cryptographic usage
- Hardcoded secrets in AI-generated code (yes, AI adds new ones)
- OWASP Top 10 pattern matching

**License compliance**
- Detect GPL/AGPL code in proprietary projects
- Flag license-incompatible dependencies
- Identify code that matches known open source

**Custom rules**
- Patterns to enforce (your team's standards)
- Patterns to forbid (banned APIs, deprecated functions)
- Framework-specific best practices
- Create rules from selected code in the extension

## How it works

Sentinel runs in three places:

### In the gateway

After the AI responds, before the code reaches your editor:

1. AI generates code
2. The gateway intercepts the response
3. Sentinel scans for vulnerabilities, license issues, and policy violations
4. Clean code passes through. Flagged code is annotated with the issue.

### In the extension

Real-time scanning as you code:

1. You save a file (or type, if on-type scanning is enabled)
2. The daemon runs Sentinel rules against your code
3. Issues appear as inline VS Code diagnostics
4. Quick-fix code actions offer remediation

### In CI/CD

On pull requests via GitHub integration:

1. A PR is opened or updated
2. The Takumo GitHub App receives the webhook
3. Sentinel scans the diff
4. Detections appear as check run annotations on the PR

## Enforcement modes

The enforcement mode depends on your policy configuration:

| Mode | Behavior |
|------|----------|
| **Block** | Flagged code is rejected. The request fails with detection details. |
| **Warn** | Code passes through with warnings attached. Logged to audit trail. |
| **Monitor** | Code passes through silently. Issue is logged. No notification. |

## Scan API

```typescript
// POST /scan
const response = await fetch('http://localhost:19532/scan', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    content: code,
    filename: 'handler.ts',
    language: 'typescript'
  })
});

const result = await response.json();
// {
//   scan_id: "scan_abc123",
//   passed: false,
//   issues: [
//     {
//       rule_id: "sql-injection",
//       severity: "critical",
//       message: "Unsanitized user input in SQL query",
//       line: 12,
//       column: 8,
//       fix_suggestion: "Use parameterized queries"
//     }
//   ],
//   engine_duration_ms: 15
// }
```

## In the dashboard

The dashboard shows Sentinel detections with:
- Detection type and severity
- Affected code location
- Suggested remediation
- Trend data over time
- Filter by repo, severity, and rule

## Custom rules

Create rules that match your team's patterns. Rules use regex or content matching with severity, category, and fix suggestions.

```json
{
  "name": "no-console-log",
  "pattern": { "type": "regex", "value": "console\\.log\\(" },
  "severity": "warning",
  "category": "code-quality",
  "message": "Remove console.log before merging",
  "fixSuggestion": "Use your logging library instead"
}
```

Rules created in the dashboard or extension sync to all team members automatically.
