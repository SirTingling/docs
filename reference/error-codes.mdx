# Error Codes

Every error response follows the same format. Parse it once, handle it everywhere.

## Response format

```json
{
  "error": {
    "type": "unauthorized",
    "message": "Authentication required"
  }
}
```

The `type` field is a stable, machine-readable snake_case string. The `message` field is human-readable. The request ID is returned in the `x-aegis-request-id` response header, not in the JSON body.

## Status codes

| Status | Type | Description |
|--------|------|-------------|
| 400 | `invalid_request` | Malformed JSON, missing required fields |
| 400 | `tenant_required` | Request is missing organization context (API key not linked to an org) |
| 400 | `unsupported_provider` | The specified AI provider is not supported |
| 401 | `unauthorized` | Invalid API key, missing credentials |
| 401 | `token_expired` | API key or JWT has expired |
| 401 | `token_revoked` | API key has been revoked |
| 403 | `forbidden` | Insufficient scope, plan limit reached |
| 404 | `not_found` | Resource doesn't exist |
| 413 | `payload_too_large` | Request body exceeds maximum size |
| 422 | `governance_violation` | Sentinel blocked the AI response (policy violation) |
| 429 | `rate_limited` | Too many requests. Check `Retry-After` header |
| 500 | `internal_error` | Something went wrong on our end. Contact support |
| 502 | `upstream_error` | Upstream AI provider returned an error |
| 503 | `too_many_concurrent` | Too many concurrent requests. Check `Retry-After` header |
| 503 | `circuit_open` | Circuit breaker tripped for upstream provider. Retry after cooldown |
| 503 | `cloud_unavailable` | Coordinator or cloud service temporarily unreachable |
| 504 | `upstream_timeout` | Upstream AI provider timed out |

## Handling 429 Rate Limited

When you hit rate limits, the response includes headers telling you when to retry:

```bash
HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1710360000
```

| Header | What it means |
|--------|--------------|
| `Retry-After` | Seconds to wait before retrying |
| `X-RateLimit-Limit` | Max requests per window |
| `X-RateLimit-Remaining` | Requests left in current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |

Back off for the number of seconds in `Retry-After`. Don't retry in a tight loop.

## Handling errors in code

```typescript
const response = await fetch("https://gateway.takumo.io/v1/shield", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${apiKey}`,
    "Content-Type": "application/json"
  },
  body: JSON.stringify({ content: code })
});

if (!response.ok) {
  const { error } = await response.json();

  switch (error.type) {
    case "unauthorized":
    case "token_expired":
    case "token_revoked":
      // Key is bad. Check config.
      break;
    case "rate_limited":
      // Wait and retry.
      const retryAfter = response.headers.get("Retry-After");
      await sleep(Number(retryAfter) * 1000);
      break;
    case "too_many_concurrent":
    case "circuit_open":
    case "cloud_unavailable":
      // Temporary overload. Wait and retry.
      const svcRetry = response.headers.get("Retry-After");
      await sleep(Number(svcRetry) * 1000);
      break;
    case "upstream_error":
      // AI provider issue, not Takumo. Check provider status.
      break;
    case "governance_violation":
      // Sentinel blocked the AI response. Review scan_id in the error.
      break;
    default:
      console.error(`Takumo error: ${error.type} - ${error.message}`);
  }
}
```

<Tip>
Always include the `x-aegis-request-id` response header value when contacting support. It maps directly to the audit trail.
</Tip>
