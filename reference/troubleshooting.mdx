# Troubleshooting

## API & Gateway

<Accordion title="401 Unauthorized">
Your API key is invalid, expired, or revoked.

**Check these:**
- Is the key correct? Copy-paste errors are common. Watch for trailing whitespace.
- Has it been revoked? Check **Settings > API Keys** in the dashboard.
- Is it for the right environment? `tk_live_` keys don't work in test mode and vice versa.
- Is the `Authorization` header formatted correctly? It should be `Bearer tk_live_...`.

```bash
# Verify your key works
curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer $TAKUMO_API_KEY" \
  https://gateway.takumo.io/v1/health
# Should return 200
```
</Accordion>

<Accordion title="403 Forbidden">
Your API key doesn't have the required scope, or your plan doesn't include the feature you're trying to use.

**Check these:**
- Key scopes: go to **Settings > API Keys**, click the key, check its scopes.
- Plan limits: you might have hit your monthly credit or request limit. Check **Settings > Billing**.
- The error response body includes the specific reason. Read the `message` field.
</Accordion>

<Accordion title="429 Rate Limited">
You've exceeded your plan's rate limit.

**What to do:**
- Check the `Retry-After` header for when to try again.
- Check `X-RateLimit-Remaining` to see your current quota.
- If this happens regularly, upgrade your plan or reduce request frequency.
- Don't retry in a tight loop. Back off exponentially.

See [Error Codes](/reference/error-codes) for the full rate limit header reference.
</Accordion>

<Accordion title="502 Bad Gateway">
The upstream AI provider (Anthropic, OpenAI) returned an error. This is not a Takumo issue.

**Check these:**
- The provider's status page ([status.anthropic.com](https://status.anthropic.com), [status.openai.com](https://status.openai.com))
- Your provider API key. Is it valid? Does it have sufficient quota?
- The error details are in the response body. Takumo forwards the provider's error message.
</Accordion>

<Accordion title="503 Service Unavailable">
The gateway or coordinator is down.

**For SaaS users:**
- Check [status.takumo.io](https://status.takumo.io) for incidents.
- Wait and retry. The gateway auto-recovers.

**For on-prem users:**
- Check your fleet status in the dashboard.
- Verify pods are running: `kubectl get pods -n takumo`.
- Check coordinator logs: `kubectl logs -n takumo deployment/gateway-coordinator`.
- Verify the coordinator's gRPC endpoint is reachable from the gateway pods.
</Accordion>

<Accordion title="Secrets not detected">
Takumo detects specific patterns. If your secret doesn't match a known pattern, it won't be tokenized.

**Check these:**
- Review [Supported Patterns](/patterns/supported) for the full list.
- Run a local scan to see what Takumo finds: `takumo-aegis scan ./your-file.ts`
- If the secret uses a non-standard format, add a custom pattern in your [policy configuration](/concepts/policies) or create a [custom Sentinel rule](/dashboard/rules).
- Generic detection catches values assigned to variables named `password`, `secret`, `api_key`, etc. If your variable name doesn't match, the value won't be flagged.
</Accordion>

<Accordion title="High latency on first request">
Normal. The first request in a session performs key exchange and vault initialization. This adds 50-200ms depending on network conditions.

Subsequent requests add approximately 3ms. If every request is slow, check your network path to the gateway. For on-prem, ensure the gateway and coordinator are in the same cluster or region.
</Accordion>

<Accordion title="Connection refused">
The gateway is unreachable.

**Check these:**
- **Gateway URL:** For SaaS, use `https://gateway.takumo.io/v1`. For on-prem, use your gateway's external URL.
- **DNS resolution:** Can your machine resolve the hostname? `nslookup gateway.takumo.io`
- **Firewall rules:** Port 443 (HTTPS) must be open to the gateway.
- **Proxy settings:** If you're behind a corporate proxy, configure it in your tool's HTTP settings.

```bash
# Test connectivity
curl -v https://gateway.takumo.io/v1/health
```
</Accordion>

<Accordion title="Gateway not appearing in fleet">
The gateway needs a valid join key to register with the coordinator.

**Check these:**
- Is the join key correct? Copy it from **Settings > Fleet > Join Keys** in the dashboard.
- Has the join key been revoked? Revoked keys are rejected immediately.
- Can the gateway reach the coordinator's gRPC endpoint? The gateway initiates the connection.
- Check gateway logs for connection errors: `kubectl logs -n takumo deployment/aegis-gateway`.
- Verify the coordinator's gRPC port (default: 9090) is accessible from the gateway's network.
</Accordion>

## Extension (Studio)

<Accordion title="Side panel shows 'Offline'">
The extension can't reach the daemon.

**Check these:**
1. Is the daemon running? Check the Takumo status bar at the bottom of VS Code.
2. Try **Takumo: Toggle Daemon** from the command palette to restart it.
3. Check the output channel: **View > Output** → select **Takumo** from the dropdown.
4. Is something else using port 19532? Change the port in settings: `takumo.daemon.port`.
5. On first install, the daemon downloads automatically. Check your network connection.
</Accordion>

<Accordion title="Daemon won't start">
**Check these:**
1. Open the Takumo output channel for error details.
2. Binary not found? The daemon auto-downloads on first activation. If behind a corporate proxy, the download may fail. Manually download from [GitHub Releases](https://github.com/SirTingling/takumo-extension/releases) and set `takumo.daemon.binaryPath`.
3. Permission denied? On macOS/Linux, the extension sets executable permission automatically. If it fails, run: `chmod +x ~/.vscode/globalStorage/takumo.takumo/daemon/takumo-daemon`
4. Checksum verification failed? The downloaded binary didn't match the expected SHA256 hash. Retry — the download may have been corrupted.
5. On macOS, you may need to allow the binary in **System Settings > Privacy & Security** on first run.
</Accordion>

<Accordion title="Sign-in fails or times out">
**Check these:**
1. The daemon may still be starting. Wait a few seconds and click **Retry**.
2. Check that cloud.takumo.io is reachable from your browser.
3. The device flow auto-expires after 10 minutes. If the browser page expired, start again from the extension.
4. If you're behind a VPN, ensure cloud.takumo.io is accessible.
5. Try signing in with an API key instead: **Takumo: Sign In with API Key**.
</Accordion>

<Accordion title="Scans not appearing">
**Check these:**
1. Is the extension enabled? Check `takumo.enabled` in settings.
2. Is the file type supported? Takumo scans 10+ file types. Check the output channel for "skipping" messages.
3. Is on-save scanning enabled? Check `takumo.scan.onSave` in settings.
4. Is the daemon running? Check the status bar.
5. Try a manual scan: **Takumo: Scan Current File** from the command palette.
</Accordion>

<Accordion title="Transform not working">
**Check these:**
1. Transforms require authentication. Sign in first.
2. Transforms are not available in demo mode.
3. Select code before running **Takumo: Transform Selection**.
4. Check the output channel for error details. The transform calls Aegis Shield, which may be temporarily unavailable.
5. Check your credit balance in the dashboard. Transforms consume credits.
</Accordion>

<Accordion title="Brain context empty or unavailable">
**Check these:**
1. Brain requires authentication. Sign in first.
2. Brain is not available in demo mode.
3. Your repo must be connected and indexed. Check the dashboard for repo connection status.
4. Indexing happens after repo connection and may take a few minutes for large repos.
5. Check the output channel for Brain-related errors.
</Accordion>

<Accordion title="Extension slowing down my editor">
Takumo's extension is designed to be lightweight, but if you notice lag:

1. Increase the debounce interval: set `takumo.scan.debounceMs` to `1000` or higher.
2. Disable on-type scanning: set `takumo.scan.onType` to `false` (keeps on-save scanning).
3. Disable clipboard monitoring: set `takumo.secrets.clipboardMonitoring` to `false`.
4. Check the output channel for repeated errors causing retry loops.
</Accordion>

## Deployment (On-Prem)

<Accordion title="NATS connection issues with Linkerd">
Linkerd's mTLS proxy can interfere with NATS TCP connections. Add the opaque-ports annotation to the NATS service:

```yaml
metadata:
  annotations:
    config.linkerd.io/opaque-ports: "4222"
```

This tells Linkerd to pass through TCP traffic on port 4222 without trying to parse it as HTTP.
</Accordion>

<Accordion title="SealedSecret not overwriting existing Secret">
If a Kubernetes Secret was created before the SealedSecret controller was installed, the controller won't overwrite it. Options:

1. Delete the existing Secret, then sync the SealedSecret.
2. Or patch the existing Secret directly and label it: `sealedsecrets.bitnami.com/managed: "true"`.

<Warning>Deleting a Secret causes downtime for any pod mounting it. Prefer patching.</Warning>
</Accordion>

<Accordion title="ImagePullBackOff on gateway pods">
The image tag in your Helm values doesn't exist in the container registry.

**Check these:**
1. Verify the image exists: `gh api /users/SirTingling/packages/container/takumo-aegis-shield/versions --jq '.[0:5]'`
2. CI only builds Docker images when gateway crate code changes (paths-filter). Daemon-only changes don't trigger builds.
3. Don't bump `image.tag` unless you've confirmed the SHA exists in GHCR.
</Accordion>
