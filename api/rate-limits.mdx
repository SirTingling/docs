Every API request is rate-limited. Limits are enforced per identifier (user ID, org ID, or IP address depending on the endpoint) using sliding window counters backed by Upstash Redis.

## Rate limiters

The Cloud API defines separate limiters for different endpoint categories. Each uses a 60-second sliding window.

| Limiter | Limit | Window | Identifier | Used by |
|---------|-------|--------|------------|---------|
| `rateLimiter` | 100 req | 60s | User ID | General API routes (notifications, dashboard endpoints) |
| `apiKeyRateLimiter` | 1,000 req | 60s | Org ID | API key-authenticated routes (digest, rules, scans) |
| `authRateLimiter` | 10 req | 60s | IP address | Auth endpoints (login, signup, password reset, device code) |
| `supportRateLimiter` | 5 req | 60s | User ID | Support request submissions |
| `salesRateLimiter` | 3 req | 60s | User ID | Sales inquiry submissions |
| `integrationRequestRateLimiter` | 3 req | 60s | User ID | Integration request form |
| `sentinelAIRateLimiter` | 5 req | 60s | Org ID | Sentinel AI review calls (LLM gateway) |
| `reportExportLimiter` | 10 req | 60s | Identifier | Report generation/export |
| `githubWebhookRateLimiter` | 100 req | 60s | Installation ID | Incoming GitHub webhook processing |
| `integrationRateLimiter` | 20 req | 60s | User ID | Integration mutations (connect, sync, rotate) |

### Sentinel AI concurrency controls

In addition to the sliding-window rate limiter, Sentinel AI reviews have Redis-backed concurrency and deduplication controls:

- **Concurrency slots**: Per-org limit on simultaneous AI reviews. Configurable per plan via `maxDailyAIReviews`. Slot TTL is 10 minutes as a safety fallback.
- **PR dedup**: 5-minute cooldown per PR per org. Prevents redundant reviews when multiple pushes happen in quick succession.

Both controls fail open on Redis errors -- a review will proceed rather than block.

## Handling rate limits

When you exceed a rate limit, the response includes a `Retry-After` header with the number of seconds to wait:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 12
```

```json
{
  "error": "Rate limit exceeded"
}
```

Implement exponential backoff for batch workloads. Do not retry in a tight loop.

```typescript
async function requestWithRetry(url: string, options: RequestInit, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fetch(url, options);

    if (response.status !== 429) return response;

    const retryAfter = Number(response.headers.get("Retry-After") ?? 30);
    await new Promise(resolve => setTimeout(resolve, retryAfter * 1000));
  }

  throw new Error("Rate limit exceeded after max retries");
}
```

## Overage behavior

- **Free plan**: Hard-capped. Overage is disabled. When you hit the monthly limit, requests are rejected until the next billing cycle.
- **Developer and Pro plans**: Overage is enabled at $0.02 per credit. You are not cut off mid-request. Overage is billed at the end of the billing cycle.
- **Enterprise plans**: Unlimited included credits. Overage is enabled at $0.00 per credit (no charge).

### Included credits per plan

| Plan | Included credits | Per-seat | Overage rate |
|------|-----------------|----------|--------------|
| Free | 1,000 | No | None (hard cap) |
| Developer | 5,000 | No | $0.02/credit |
| Pro | 5,000 | Yes (per seat) | $0.02/credit |
| Enterprise | Unlimited | No | $0.00/credit |

Pro plans multiply the credit allowance by seat count. A Pro org with 10 seats gets 50,000 included credits per month.

<Tip>If you consistently hit rate limits, check your plan's monthly allowance first. Per-minute limits reset quickly, but monthly limits do not. Upgrading your plan is usually more cost-effective than optimizing around limits.</Tip>
