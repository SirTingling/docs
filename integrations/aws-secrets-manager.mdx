<Note>Requires an Enterprise plan.</Note>

Connect AWS Secrets Manager to map detected AWS secrets to their managed entries. When Takumo finds an AWS credential in your code, it checks Secrets Manager to identify the source secret -- enabling faster rotation and remediation.

---

## Setup

<Steps>
  <Step title="Create an IAM role">
    Create an IAM role (e.g. `TakumoSecretsReadOnly`) with a custom trust policy.
  </Step>
  <Step title="Set the External ID">
    In the trust policy, set the External ID condition to the value pre-filled in the Takumo connection form. This value is unique to your Takumo org.
  </Step>
  <Step title="Attach permissions">
    Attach an inline permissions policy with `secretsmanager:ListSecrets` and `secretsmanager:DescribeSecret` (Resource: `"*"`).

    <a href="#iam-setup">See full IAM setup below.</a>
  </Step>
  <Step title="Connect in Takumo">
    Paste the Role ARN in **Dashboard > Settings > Integrations > AWS Secrets Manager** and click **Connect**. Takumo will assume the role to verify access.
  </Step>
</Steps>

## Credential fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| **AWS Region** | Text | Yes | The AWS region (e.g. `us-east-1`) |
| **Authentication** | Select | Yes | `STS AssumeRole (recommended)` or `Static Access Key` |

### STS AssumeRole fields (visible when Authentication = STS AssumeRole)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| **Role ARN** | Text | Yes | The IAM role Takumo will assume (e.g. `arn:aws:iam::123456789012:role/TakumoReadOnly`) |
| **External ID** | Text | Yes | Pre-generated for your org. Copy this into the IAM trust policy |
| **Session Name** | Text | No | Defaults to `takumo-secrets-sync` |

### Static Access Key fields (visible when Authentication = Static Access Key)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| **Access Key ID** | Text | Yes | IAM access key (e.g. `AKIAIOSFODNN7EXAMPLE`) |
| **Secret Access Key** | Password | Yes | IAM secret key |

### Optional

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| **KMS Key Filter** | Text | No | Only sync secrets encrypted by this KMS key (e.g. `arn:aws:kms:us-east-1:123456789012:key/...`) |

## IAM setup

### Trust policy

Replace `EXTERNAL_ID` with the value shown in the Takumo connection form:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::TAKUMO_ACCOUNT_ID:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "EXTERNAL_ID"
        }
      }
    }
  ]
}
```

### Permissions policy

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "secretsmanager:ListSecrets",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "*"
    }
  ]
}
```

Takumo only needs read and list access. It never modifies, creates, or deletes secrets.

---

## Authentication

### STS AssumeRole (recommended)

STS AssumeRole is the recommended approach. Takumo assumes the IAM role using the External ID as an additional security condition. This avoids storing long-lived AWS credentials and follows AWS best practices for cross-account access.

### Static Access Key

For environments where cross-account role assumption isn't possible, you can provide an IAM access key and secret key directly. The IAM user must have the same `secretsmanager:ListSecrets` and `secretsmanager:DescribeSecret` permissions. Static keys are less secure because they are long-lived and must be rotated manually.

---

## Health check

The AWS Secrets Manager integration uses a custom health check (not the generic catalog runner). Takumo authenticates with the provided credentials and calls `secretsmanager:ListSecrets` with `MaxResults=1`. A successful response confirms:

- The credentials are valid
- The role can be assumed (for STS)
- Secrets Manager is accessible in the specified region

---

## How correlation works

When Takumo detects an AWS secret in your code:

1. Extracts the credential fingerprint (key ID, secret prefix, etc.)
2. Queries the connected Secrets Manager for matching entries
3. If a match is found, links the detection to the managed secret
4. Provides rotation guidance with a direct link to the AWS console

If a **KMS Key Filter** is configured, only secrets encrypted with that specific KMS key are considered during correlation.
