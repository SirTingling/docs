Complete reference for configuring all Takumo services. All values are set via Helm chart values, which generate Kubernetes ConfigMaps with the corresponding environment variables.

## Gateway (Aegis Shield)

### Core settings

| Environment Variable | Helm Value | Default | Description |
|---------------------|-----------|---------|-------------|
| `TAKUMO_LISTEN_ADDR` | `server.port` | `0.0.0.0:8080` | Address and port the gateway listens on |
| `TAKUMO_SHUTDOWN_TIMEOUT` | `server.shutdownTimeout` | `30` | Graceful shutdown timeout in seconds |
| `TAKUMO_AUTH_MODE` | `auth.mode` | `jwt` | Authentication mode (`jwt` or `none`) |
| `TAKUMO_AUTH_ISSUER` | `auth.issuer` | `https://cloud.takumo.io` | Expected JWT issuer |
| `TAKUMO_AUTH_AUDIENCE` | `auth.audience` | `aegis-shield` | Expected JWT audience |
| `TAKUMO_AUTH_PUBLIC_KEY_PATH` | `secrets.jwt.key` | - | Path to JWT public key (mounted at `/etc/takumo/keys/`). Only set when `secrets.jwt.enabled` is `true`. |
| `TAKUMO_FAIL_MODE` | `aegis.failMode` | `fail_closed` | Behavior when coordinator is unreachable: `fail_closed` rejects requests, `fail_open` passes them through |
| `TAKUMO_MAX_VAULT_ENTRIES` | `aegis.maxVaultEntries` | `10000` | Maximum secrets held in the in-memory vault |
| `TAKUMO_MAX_CONCURRENT_SESSIONS` | `aegis.maxConcurrentSessions` | `5000` | Maximum concurrent active sessions |
| `TAKUMO_AUDIT_BUFFER_CAPACITY` | `auditBuffer.capacity` | `50000` | In-memory audit event buffer capacity before flush |

## NATS

NATS configuration is rendered into a `nats.conf` file (not environment variables) via a ConfigMap.

| Setting | Helm Value | Default | Description |
|---------|-----------|---------|-------------|
| JetStream enabled | `jetstream.enabled` | `true` | Required for durable event delivery |
| Max memory | `jetstream.maxMemory` | `1Gi` | JetStream memory limit |
| Max file storage | `jetstream.maxFile` | `45Gi` | JetStream file storage limit |
| Client port | `ports.client` | `4222` | NATS client connection port |
| Cluster port | `ports.cluster` | `6222` | NATS inter-node cluster port |
| Monitor port | `ports.monitor` | `8222` | NATS HTTP monitoring port |
| Persistence size | `persistence.size` | `50Gi` | PVC size for JetStream data |
| Replica count | `replicaCount` | `3` | StatefulSet replicas (cluster auto-configured when >1) |

### NATS authentication

NATS uses NKey authentication. The full `authorization { ... }` block is stored in a SealedSecret and included into `nats.conf` at runtime. Four NKey users per cluster:

| Key | Used by | Purpose |
|-----|---------|---------|
| `stream-ingest` | Stream Ingest | Publish events |
| `stream-router` | Stream Router | Subscribe to events, route to destinations |
| `coordinator` | Gateway Coordinator | Publish detection events |
| `worker` | Cloud Worker | Subscribe to detection events |

<Warning>For Linkerd service mesh: add `config.linkerd.io/opaque-ports: "4222"` annotation to the NATS service. Linkerd's mTLS proxy can interfere with NATS TCP connections.</Warning>

## Secrets management

The Helm charts support three secrets providers, controlled by `secrets.provider`:

<Accordion title="Sealed Secrets (default)">
  Uses [Bitnami Sealed Secrets](https://github.com/bitnami-labs/sealed-secrets) for encrypting secrets in Git. Set `encryptedData` fields in your values file with the output of `kubeseal`.

  ```yaml
  secrets:
    provider: sealed
    database:
      enabled: true
      secretName: coordinator-database
      encryptedData: "AgBy3i4OJSWK+PiTySYZZA9rO..."
  ```

  This is the recommended approach for production environments.
</Accordion>

<Accordion title="External Secrets">
  Uses the [External Secrets Operator](https://external-secrets.io/) for pulling secrets from Vault, AWS Secrets Manager, GCP Secret Manager, or Azure Key Vault.

  ```yaml
  secrets:
    provider: external
  ```

  You manage the ExternalSecret resources separately.
</Accordion>

<Accordion title="Manual">
  You create Kubernetes secrets directly. This is the default for on-prem deployments.

  ```yaml
  secrets:
    provider: manual
    database:
      enabled: true
      secretName: coordinator-database
      key: database-url
      directKey: database-url-direct
  ```

  Create the secrets with `kubectl create secret` before installing the Helm chart.
</Accordion>

## Autoscaling

Most charts include HPA configuration. The scale-down policy is consistent across all charts (25% per minute, 5-minute stabilization window). Scale-up allows 100% per 30 seconds or 4 pods per 60 seconds. However, each service has different replica counts and CPU targets tuned for its workload:

| Service | minReplicas | maxReplicas | CPU target | Memory target |
|---------|------------|------------|------------|---------------|
| Gateway (Aegis Shield) | 2 | 10 | 70% | 80% |
| Coordinator | 2 | 10 | 70% | 80% |
| Stream Ingest | 2 | 10 | 70% | 80% |
| Stream Router | 2 | 20 | 70% | 80% |
| Vault Service | 2 | 10 | 70% | 80% |
| Sentinel Service | 2 | 10 | 60% | - |
| Brain Indexer | 3 | 20 | 60% | - |
| Brain Embedding | 4 | 12 | 60% | - |
| Brain Query | 4 | 16 | 70% | - |

<Note>NATS runs as a StatefulSet (default 3 replicas) and brain-pattern runs as a CronJob — neither uses HPA.</Note>

## Pod security

All charts enforce a restrictive security context by default: read-only root filesystem, no privilege escalation, all Linux capabilities dropped, and seccomp set to `RuntimeDefault`. The user/group IDs vary by base image:

| Service | UID/GID | Base image |
|---------|---------|------------|
| Gateway, Coordinator, Vault, Sentinel, Brain Indexer, Brain Embedding, Brain Query, Brain Pattern | 65534 (`nobody`) | Standard |
| Stream Ingest, Stream Router | 65532 | `distroless/cc-debian12:nonroot` |
| NATS | 1000 (`nats`) | `nats:2.10-alpine` |

```yaml
# Common security context (UIDs differ per service — see table above)
podSecurityContext:
  runAsNonRoot: true
  runAsUser: 65534   # or 65532 / 1000 depending on service
  runAsGroup: 65534
  fsGroup: 65534
  seccompProfile:
    type: RuntimeDefault

containerSecurityContext:
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL
```
